> 2026-03-10 작성/갱신. 시스템 분석 + 설계 방향 논의에서 도출.

## 정체성 정의

**ENOMETA = 대본 기반 제네레이티브 미디어아트**

- 대본(스토리 구조)이 음악과 비주얼의 뼈대를 **생성**한다
- 오디오/SI가 생성된 구조 위에서 **반응**한다
- 리액티브는 효과이지, 본질이 아니다

```
제네레이티브 미디어아트
 │
 ├─ 제네레이티브 (뼈대 생성) ← 상위
 │   입력: 대본 스토리 구조
 │   출력: 음악 섹션/레이어/아크, 비주얼 장면/형상/vocab
 │   시점: 렌더링 전 1회
 │   담당: Python 기본 자동 + Claude 의미 기반 오버라이드
 │
 └─ 리액티브 (표면 변조) ← 하위
     입력: SI, 오디오 프레임 (onset/bass/rms)
     출력: 음색, 드라이브, 밝기, 속도, 떨림
     시점: 매 프레임
     담당: 기존 코드 유지
```

> **"뭘 만들지"를 대본이 결정하고, "어떻게 움직이지"를 오디오/SI가 결정한다.**

## 설계 원칙: 제네레이티브 → 리액티브 2레이어

```
대본 (완성물)
 │
 ├─ 스토리 구조 (서사, 전환점, 문단 흐름)
 │   → 음악 뼈대 설계        [제네레이티브]
 │   → 비주얼 형상/장면 설계  [제네레이티브]
 │
 └─ SI (감정 강도, 0.05~1.0)
     → 음악 표현 변조 (음색, 드라이브)  [리액티브]
     → 비주얼 반응 (밝기, 레이어 수)    [리액티브]
     → audio → 비주얼 프레임 반응       [리액티브] ← 이미 강함, 유지
```

### 왜 이 구조인가

| 문제 | 원인 | 해결 |
|------|------|------|
| 음악이 대본과 약하게 연결됨 | SI 하나로 모든 걸 처리 (볼륨 ±5%) | 스토리→구조 설계 + SI→표현 변조, 역할 분리 |
| SI가 음악 완성도를 해침 | 감정 데이터가 음악 구조에 직접 간섭 | SI는 표면(음색/드라이브)만. 구조는 건드리지 않음 |
| 비주얼이 추상적 | 리액티브만 있고 형상 생성 없음 | 스토리→형상 생성(제네레이티브) + audio→반응(리액티브) |
| 같은 뼈대 위 표면 변화만 | 고정 ARRANGEMENT_TABLE + 장르→전략 매핑 | 매번 다른 뼈대 생성 → 같은 리액티브가 다른 결과 |

### 핵심 규칙

1. **제네레이티브가 먼저**: 스토리 구조가 음악/비주얼의 뼈대를 설계한다
2. **리액티브는 표면만**: SI/audio는 완성된 구조를 해치지 않고 표현만 변조한다
3. **음악의 자기 문법 보호**: 뼈대 안에서 음악은 4바/8바 문법으로 자유롭게 완성된다
4. **기존 자산 전부 유지**: vocab 30종, Python 레이어 10종, SI 추출 — 선택/조립 방식만 변경

## 파이프라인 변화

```
현재:  대본 → TTS → SI 추출 → 음악(SI 반응) → 비주얼(장르→랜덤 vocab)
                     ↑ 모든 게 SI에서 시작

변경:  대본 → 스토리 구조 분석 → Claude 설계안 → 음악 뼈대 생성 + 비주얼 장면 생성
        │                                              ↓                    ↓
        └→ TTS → SI 추출 → 오디오 ──────────→ 리액티브 변조 (기존 코드)
```

### Claude 개입 시점

| 단계 | 현재 | 제네레이티브 후 |
|------|------|----------------|
| 대본 작성 | Claude | 동일 |
| 대본→데이터 추출 | 자동 | 자동 (+ 스토리 구조 추출 추가) |
| 음악/비주얼 설계 | 자동 (고정 규칙) | **하이브리드**: Python 기본 자동 + Claude 오버라이드 |
| 리액티브 | 자동 | 동일 |

> 설계안은 대본 컨펌 직후 **같은 세션에서** 제시 (추가 비용 = 메시지 2~3개)

## 구현 로드맵

### Phase 0: 스토리 구조 추출 (공통 기반)

- `script_data_extractor.py` 확장: 문단/전환점/에너지 아크 추출
- 기존 SI + 키워드/품사 유지, 스토리 구조 필드 추가
- 이 단계가 Phase 1A/1B의 공통 의존성

### Phase 1A: 스토리→음악 제네레이티브

```
현재: ep_seed + 장르 스펙 → ARRANGEMENT_TABLE (고정)
     SI → 볼륨 ±5%, 게이트, 음색

목표: 스토리 구조 → ARRANGEMENT_TABLE 동적 생성
     SI → 음색/드라이브 변조만 (구조 비간섭)
```

**스토리가 결정하는 것:**
- 섹션 전환 시점 (문단 경계 → 가장 가까운 4바 snap)
- 레이어 ON/OFF 시점 (스토리 에너지 곡선 기반)
- 에너지 아크 (intro→build→climax→resolution 매핑)
- 장르 내 변주 방향 (ep_seed와 협력)

**스토리가 결정하지 않는 것 (기존 유지):**
- BPM (장르 범위 내 고정)
- 드럼 패턴 (seq_config)
- 멜로디 음정 (Norgard 수열 + ep_seed)
- 음색 디테일 (SI 리액티브 담당)

### Phase 1B: 스토리→비주얼 제네레이티브 (1A와 병렬 가능)

```
현재: 장르→전략 고정 매핑, vocab 랜덤 배치
     결과: 추상 패턴만. 인식 가능한 형상 없음

목표: 스토리 의미 → vocab/형상 선택 + 장면 설계
     SI + audio → 생성된 형상이 반응 (리액티브)
```

**스토리가 결정하는 것:**
- vocab 선택 (문단 키워드/의미 → 맥락 맞는 vocab)
- 장면 밀도 차등 배치 (스토리 흐름 기반)
- 비주얼 전략 전환 (장면별로 다른 전략 가능)
- 형상 생성 (주제 기반 비트맵/실루엣/ASCII)

**형상 구현 3가지 방식:**

| 방식 | 설명 | 기존 자산 |
|------|------|----------|
| 형상 사전 | Pillow 비트맵 2D 배열, 키워드→형상 매핑 | pixel_grid 확장 |
| 실루엣 마스크 | CC0 실루엣 PNG → ASCII 문자 채움 | ascii_block 확장 |
| 수학적 절차 생성 | 나선, 프랙탈, 파동 간섭 등 수식 기반 | lissajous 확장 |

**절대 원칙: AI 이미지 생성 금지.** 코드가 직접 그리는 거칠고 원초적인 형상만 사용.

**형상 축적 전략:**
- 에피소드 제작 시 주제에 필요한 형상을 만들어 사전에 추가
- EP가 쌓일수록 형상 라이브러리 자동 확장 (EP030쯤 50~60개 예상)
- 이전 형상을 다른 맥락에서 재활용 가능

**비주얼 레이어 스택:**

```
Layer 3: 리액티브 vocab   ← Remotion (매 프레임 오디오 반응)
         TextReveal, Lissajous, AsciiArt 등 30종
         PostProcess (CRT, 스캔라인, 글리치)

Layer 2: 형상화 (신규)    ← Python (렌더링 전 1회 생성)
         비트맵 스프라이트, ASCII 실루엣 마스크
         대본 키워드 → 형상 사전에서 선택

Layer 1: 배경 (현행 유지) ← Python 10종
         SineWave, Waveform, Particle, TextData,
         Barcode, DataStream, DataMatrix, Feedback,
         Bytebeat, AsciiBackground
```

> 배경 10종은 현행 유지. 형상화 레이어 추가로 배경 역할이 축소되므로 추가 불필요.

### Phase 2: SI 리액티브 재정의

- Phase 1A/1B 완료 후 자연스럽게 따라옴
- SI 역할: 구조 결정권 제거, 표면 변조 전담
- 코드 변경 최소 — SI가 건드리던 구조 결정 부분만 제거
