# ENOMETA 창작 자유도 변경 보고서

> 작성일: 2026-03-04
> 성격: 변경 보고서 — 감사 결과를 **유지 / 변경 / 완화 / 보류** 4단계로 분류

## 배경

EP005~EP007 피드백을 반영하면서 "안전성" 규칙이 과도하게 쌓였다.
시스템이 "나쁜 결과를 방지"하려다 "새로운 시도 자체를 방지"하는 상태.

**이 보고서의 목표**: 꼭 지켜야 할 것은 지키되, 불필요한 족쇄를 풀어준다.

**핵심 원칙**:
- 자막(SubtitleSection) = 가독성 도구 → 스타일 고정
- 비주얼 텍스트(TextReveal) = 모션그래픽 → 자막과 확실히 다르게
- 음악 = ikeda 카피가 아닌 **대본이 조종하는 리액티브 댄스 뮤직**
- 음악적 문법 존중 (빌드/드롭, 긴장/이완, 패턴/루핑)
- raw 컴퓨팅 바이브 (원초적이되, 드럼 구조 위에서)

## B. 변경 — 비주얼 개선 (6건)

영상의 시각적 부분만 건드리는 변경. 음악은 F 섹션에서 별도 처리.

### B-1. 자막 vs 비주얼 텍스트 확실히 구분

지금은 둘 다 화면에 글자가 뜨는 건데, 시청자가 헷갈릴 수 있음.

| | 자막 (SubtitleSection) | 비주얼 텍스트 (TextReveal) |
|---|---|---|
| 역할 | 대본 읽기 보조 | 시각 효과 |
| 위치 | 하단 고정 (y≈1370) | 자유 (자막 영역 y≥1230 만 침범 금지) |
| 색상 | 흰색 | 감정별 컬러 |
| 움직임 | 페이드만 | wave/glitch/scatter 등 |
| 배경 | 반투명 검정 | 없음 |

**핵심**: 자막은 안 건드림. TextReveal만 더 차별화.

### B-2. ShapeMotion 파라미터 유연화

지금은 도형 애니메이션 속도/크기가 코드에 박혀있음 (회전 45도/s, 펄스 3Hz 등).
→ visual_script에서 `speed`, `scale`, `count`를 에피소드마다 다르게 줄 수 있게.

### B-3. Python 레이어 강도를 SI에 연동

지금은 배경 레이어 강도가 0.7/0.4/0.3으로 고정.
→ SI(감정 강도)에 따라 ±20% 변동 허용. 조용한 장면은 약하게, 강한 장면은 세게.

### B-4. 감정→비주얼 매핑 확장

지금은 11개 감정 × 6개 비주얼 효과 = 닫힌 풀. 배경도 calm/turbulent 2개뿐.
→ 각 감정에 대안 효과 2~3개 추가, 배경 모드도 +subtle/intense.

### B-5. 엔드카드 태그라인 동적화

지금은 "존재와 사유, 그 경계를 초월하다"가 매번 똑같이 나옴.
→ 에피소드마다 다른 태그라인 가능. 안 주면 기존 문구 자동 사용.

### B-6. highlight_words 확장

지금은 2~4글자 명사만, 최대 3개 자동 추출.
→ 동사/형용사도 가능, 최대 5개, 수동 지정도 가능.

## C. 완화 — 과도한 규칙 풀기 (5건)

코드 수정 없음. 문서와 설정 파일의 "금지/절대" 용어를 부드럽게.

### C-1. feedback_defaults.json 용어 전환

`"반드시"` → `"기본값"` / `"금지"` → `"비권장"` / `"절대"` → `"주의"`
(단, `py` 명령이나 Edge-TTS 같은 기술적 필수사항은 "절대" 유지)

### C-2. "순수 노이즈 금지" 완화

지금은 노이즈 계열 소리를 완전 차단해놨음.
→ "리듬 구조 위에서 사용 권장, 의도적이면 허용"

### C-3. bytebeat/feedback "최소화" 삭제

지금은 "bytebeat: 최소화", "feedback: 최소화"로 억눌러놨음.
→ 볼륨 상한(0.3)만 유지. "최소화하라"는 지시 자체를 삭제.

### C-4. TextReveal bottom 금지 완화

지금은 화면 하단에 비주얼 텍스트 놓는 걸 완전 금지.
→ 자막 영역(y≥1230)과 안 겹치면 허용.

### C-5. MEMORY.md 자기확증 수정

"ikeda 장르 다양성 점검 결과: 제약 아님"이라고 적혀있는데, 이건 자기합리화.
→ "패턴 엔진 전환 예정. 현재 엔진은 BPM/키 범위에 제한 있음"으로 교체.

## F. 음악 엔진 재설계 — 대본 리액티브 댄스 뮤직

### 방향

ENOMETA의 음악은 **누군가의 스타일을 카피하는 게 아니다**.
대본 데이터(SI, 감정, 타이밍)가 음악의 파라미터를 실시간으로 조종해서,
**비주얼-음악-나레이션이 하나의 유기체로 반응**하는 것이 목표.

**미학적 지향**:
- **raw 컴퓨팅 바이브** — 컴퓨팅 특유의 원초적이고 거친 전자음
- **음악적 문법** — 데이터→소리 변환이 아님. 빌드/드롭, 긴장/이완, 호흡이 있는 진짜 음악
- **댄스 뮤직** — 패턴과 루핑이 핵심. 들어갈 땐 들어가고 빠질 땐 빠지는 흐름

### 현재 엔진의 문제

지금 음악 엔진은 **"토막을 만들어서 이어붙이는"** 구조.
대본의 각 씬(섹션)마다 소리를 따로 만들고 순서대로 갖다 붙인다.
그래서 섹션이 바뀔 때마다 **"다른 곡이 시작되는"** 느낌이 남.

```
섹션 A                    섹션 B
[킥 + 톱니파 + 글리치]    [아르페지오 + 드론 + 산란]
     페이드아웃 ─┐  ┌─ 페이드인
                │갭│
                └──┘
                ↑ 여기서 "곡이 바뀐" 느낌
```

구체적 원인 9가지:

| # | 원인 | 증상 |
|---|------|------|
| 1 | 전환 데이터를 만들어놓고 안 씀 | 크로스페이드 없음 |
| 2 | 텍스처를 섹션마다 따로 렌더 | 섹션 간 겹침 없음 |
| 3 | 스무딩이 계단+이동평균 | 진짜 부드러운 전환이 아님 |
| 4 | 전체 볼륨을 일괄 곱셈 | 에너지가 갑자기 확 바뀜 |
| 5 | 각 소리가 독립적으로 페이드 | 페이드 사이에 미세한 침묵 |
| 6 | 마디 경계 맞추기가 나레이션과 불일치 | 음악 끊김 ≠ 대사 끊김 |
| 7 | 악기가 on/off 이진 전환 | 갑자기 사라지고 갑자기 등장 |
| 8 | 밀도와 볼륨이 동시에 점프 | 경계에서 이중 불연속 |
| 9 | 에너지 곡선이 타임라인 %로 계산 | 실제 감정과 안 맞음 |

### 새 엔진의 핵심 설계

#### 1. 드럼 패턴이 뼈대

댄스 음악에서 **킥/하이햇/스네어**는 곡 전체를 관통하는 골격.
지금은 리듬을 "여러 소리 중 하나"로 취급하는데, 새 엔진에서는 **드럼이 1순위**.

```
킥:    [x . . . x . . . x . . . x . . .]   ← 4/4 기본
하이햇: [. x . x . x . x . x . x . x . x]   ← 오프비트
스네어: [. . . . x . . . . . . . x . . .]   ← 2, 4박
```

- SI 높으면 → 하이햇 16분음표로 촘촘하게, 킥 변형, 스네어 롤
- SI 낮으면 → 킥만 남기거나 하이햇 간소화
- 감정 전환 → 필인(fill) 넣고 새 패턴
- **빌드업**: 요소 하나씩 추가
- **드롭**: 한꺼번에 투입
- **브레이크**: 킥 빼기

#### 2. 음악적 문법

소리를 시간 위에 배치하는 방법. "데이터를 소리로 변환"하는 게 아니라 **음악답게** 만드는 구조:

| 문법 | 지금 | 새 엔진 |
|------|------|--------|
| 빌드/드롭 | 없음 | 4~8마디 빌드 → 필인 → 드롭 |
| 호흡 | 끊김 | 의도적 1~2박 브레이크 |
| 반복과 변주 | 섹션마다 다른 소리 | 코어 루프 유지 + 4마디마다 변주 |
| 긴장/이완 | 볼륨만 변화 | 하이햇 밀도↑, 필터 스윕, 스네어 롤 → 해소 |
| 콜 앤 리스폰스 | 없음 | 킥 ↔ 멜로디 프레이즈 교대 |

#### 3. 패턴 + 루핑

[TidalCycles](https://tidalcycles.org/)의 "순환 시간" 개념을 빌려옴.
**토막 이어붙이기 대신, 하나의 루프가 계속 돌면서 점진적으로 변형.**

| 지금 | 새 엔진 |
|------|--------|
| 섹션 A → B → C 이어붙이기 | 코어 루프가 점진적 변형 (하나의 곡) |
| 악기 on/off | 파라미터가 연속적으로 모핑 |
| 섹션마다 다른 악기 | **모든 레이어 항상 존재**, 볼륨/밀도만 변화 |
| 마디 경계에서 끊김 | 사이클 단위 반복 → 경계 없음 |

#### 4. raw 컴퓨팅 바이브

세련된 프로덕션이 아니라 **코드가 직접 소리를 뿜는** 느낌:
- sawtooth, square, bytebeat, 피드백 루프
- 비트크러싱, 샘플레이트 감소
- 글리치, 스터터, 게이트 시퀀서
- **단, 전부 드럼 그리드 위에 정렬** — 무질서가 아니라 raw한 질서

#### 5. 대본이 음악을 조종

| 대본 데이터 | 음악에서 뭐가 바뀌나 |
|------------|-------------------|
| SI (감정 강도) | 드럼 밀도, 레이어 수, 필터 열림/닫힘 |
| 감정 (tension→awakening) | 패턴 변형 방향, 빌드/드롭 타이밍 |
| 씬 전환 | 드럼 필인 위치 |
| highlight_words | 음악적 악센트 (킥+스네어 동시 치기, 브레이크→드롭) |

### 장르 = 프리셋

| 지금 | 새 엔진 |
|------|--------|
| `if genre == "ikeda":` (3000줄) | JSON 프리셋 (30줄) |
| 장르 추가 = 대규모 개발 | 장르 추가 = 프리셋 파일 하나 추가 |
| BPM/키 변경 = 하드코딩 수정 | BPM/키 = 파라미터 |

장르는 "코드 분기"가 아니라 **드럼 패턴 + BPM + 스케일 + raw 레벨의 조합**.
새 장르나 키 추가는 엔진 재설계의 자연스러운 부산물.

### 기존 음악 이슈 → 전부 흡수

개별적으로 수정하려던 항목들이 패턴 엔진에 자동으로 해결됨:
- ~~SI 변화폭 조절~~ → 연속 파라미터 곡선
