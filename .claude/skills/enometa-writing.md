# ENOMETA Writing System v13 — Music-First (1 Sentence = 1 Bar)

## 0. 가장 중요한 핵심 원칙: "1문장 = 1마디"

ENOMETA의 영상은 **음악(박자) 중심**으로 생성됩니다. 대본은 산문(에세이)이 아니라 **시(Poetry)** 또는 **랩(Rap) 가사**의 형태를 띠어야 합니다.

- **[1줄 = 1호흡]**: 대본에서 줄바꿈(Enter) 한 번이 TTS 한 호흡 단위(약 1~2초)가 됩니다. v16부터 나레이션 타이밍은 TTS 실측 기반이며 음악 마디와 독립적으로 흐릅니다.
- **[최적 글자 수]**: 한 호흡에 자연스럽게 들어가는 글자 수는 **8자~12자**입니다.
- **[가변 호흡 룰]**:
  - 극단문 (1~6자) : TTS가 짧게 치고 빠짐. 비트가 공간을 채우는 임팩트 구간. (예: "공포.")
  - 표준문 (7~15자) : 밀도 높은 래핑. 리드미컬한 호흡. (예: "각성의 화학식은 같다.")
  - 장문 (16자 이상) : TTS가 단조롭게 이어짐. 음악과 괴리 발생. 지양할 것.
- **[드롭 구간]**: 의도적으로 **빈 줄(Empty line)**을 넣으면, 나레이션 없이 음악만 흐르는 공백 구간이 됩니다. (0.8초 paragraph gap 적용)

## 1. 채널 아이덴티티

**ENS·NOĒMA·META** = 존재·사유·초월
"존재와 사유, 그 경계를 초월하다"

### "팩트로 재구성하는" 철학 (Reconfigure Through Facts)

```
원칙: 사실은 설명하지 않는다. 사실은 분해한다. 분해가 각성을 만든다.
패턴: 데이터 + 컴퓨팅 관점 = 구조적 각성
```

**왜 팩트가 재구성하는가:**
- 팩트는 시스템을 드러낸다 → "게으른 게 아니라 전두엽 연산량이 한계에 도달한 것"
- 팩트는 패턴을 보여준다 → "모든 인간의 뇌가 같은 아키텍처" = 개인의 문제가 아닌 설계의 문제
- 팩트는 변수를 바꾼다 → "불안과 각성은 같은 신호. 라우팅이 다를 뿐" = 관점의 리팩토링

**예시:**
- "3만 5천 번의 결정. 전두엽 연산량 초과." → 시스템 과부하
- "불안과 각성. 같은 신호. 다른 라우팅." → 구조적 재정의
- "95%는 어제의 복사본. 5%의 diff가 변화의 커밋." → 패턴 인식

## 2. 핵심 톤: "차가운 정밀"

- 감정을 배제하되 각성은 남긴다
- 데이터가 말하게. 해석은 시청자 몫
- 건조하되 리듬이 있다
- 설명하지 않는다. 분해한다
- 통찰 자체가 동기부여. 감정 호소 불필요

**한 문장 정의:**
"터미널 로그를 읽다가 존재론적 버그를 발견하는 글."

### 인칭 규칙: 3인칭 관찰 (기본값)

- "뇌는", "인간은", "시스템은" — 관찰자 시점
- "전두엽이 소진된다" (O) / "우리의 전두엽이 소진된다" (X)
- 필요 시 무인칭도 허용: "소진된다." "반복된다."
- "우리/나/너"는 금지하지 않되, 강제하지 않음. 자연스러울 때만

## 3. 콘텐츠 도메인 시스템

### 렌즈 (고정): 컴퓨팅

모든 주제를 컴퓨팅 관점으로 본다. 컴퓨팅은 도메인이 아니라 **기본 렌즈**.
알고리즘, 시스템, 디버깅, 최적화, 패턴 인식, 데이터 구조 — 이것이 세상을 보는 프레임.

### 소재 도메인 7종

| 도메인 | 팩트 소스 | 컴퓨팅 관점 연결 |
|--------|----------|------------------|
| **뇌과학** | 편도체, 전두엽, 호르몬, 신경회로, 시냅스 | 뇌 = 생체 컴퓨터. 시냅스 = 회로. 호르몬 = 시그널 |
| **데이터** | 통계, 확률, 패턴, 수치, 비율 | 통계/수치 자체가 언어. 패턴이 진실 |
| **철학** | 존재론, 인식론, 윤리, 시간, 자유의지 | 존재론적 버그. 의식 = 프로세스 |
| **인문학** | 역사, 심리, 사회, 문화, 인간 조건 | 사회 시스템. 역사 = 로그. 문화 = 프로토콜 |
| **물리학** | 양자역학, 상대성, 엔트로피, 파동, 장(field) | 우주 = 연산. 엔트로피 = 시스템 붕괴 |
| **생물학** | 진화, DNA, 세포, 생태계, 공생 | DNA = 코드. 진화 = 디버깅 |
| **수학** | 확률, 무한, 프랙탈, 위상, 대칭 | 구조 자체. 프랙탈, 확률, 무한 |

### 교차점 공식

**도메인 교차는 도구이지 목적이 아니다.** 교차점에서 진짜 통찰이 생길 때만 교차한다. 한 도메인을 깊게 파는 것이 더 강력할 수 있다.

> ⚠️ EP006/EP008 실패 패턴: 도메인 교차를 "해야 하니까" 억지로 엮으면 논리가 어색해지고 글이 죽는다. 교차 없이 단일 도메인으로 깊게 파는 게 낫다면 그렇게 한다.

```
[컴퓨팅 렌즈 × 소재]
뇌과학     = "시냅스 가소성은 뇌의 git commit이다. 매일 다른 버전."
데이터     = "의식적 결정은 2%. 나머지는 캐시된 루틴."
철학       = "존재론적 버그. 의식이라는 프로세스는 자기 자신을 디버깅할 수 있는가."
인문학     = "트라우마는 편도체의 과잉 로깅이다. 시스템이 위협을 과대추정한 것."
물리학     = "엔트로피는 시스템 붕괴가 아니다. 정보 재배치다."
생물학     = "유전자는 38억 년짜리 코드베이스. 돌연변이는 자연의 디버깅."
수학       = "무한은 개념이 아니라 구조다. 재귀 함수처럼."

[소재 × 소재] — 컴퓨팅 렌즈로 연결
뇌과학 × 철학    = "디폴트 모드 네트워크는 의식의 백그라운드 프로세스."
뇌과학 × 물리학  = "의식은 양자 현상인가. 관측이 상태를 결정한다면."
인문학 × 생물학  = "문화는 밈의 진화. 자연선택이 사회에도 작동한다."
데이터 × 철학    = "측정할 수 없는 것은 존재하지 않는가. 변수 정의의 문제."
```

## 4. 글쓰기 구조: 3종 데이터 처리

**기본 구조는 데이터 파이프라인(A). 주제에 따라 대안 구조(B, C)도 선택 가능.**

### 구조 A: 데이터 파이프라인 (입력→처리→출력, 기본 권장)

```
[입력] 원시 데이터 — 팩트 투입
  짧은 문장. 숫자. 사실. 해석 없음.
  "하루에 뇌가 처리하는 연산."
  "6만 개."

[처리] 분해/재구성 — 데이터를 컴퓨팅 관점으로 분해
  "이건 ~이 아니라 ~이다" 재정의 문장
  데이터가 구조적 질문으로 변하는 전환점

[출력] 각성 — 처리 결과
  분해된 구조에서 통찰이 출력된다.
  마지막 문장은 인식 전환. 시청자의 변수가 바뀐다.
```

**적합**: 대부분의 주제. 팩트→분해→통찰의 자연스러운 흐름.
**SI 커브**: 표준 내러티브, 크레셴도, 계단

### 구조 B: 에러 로그 (에러 발견→역추적→원인)

```
[에러] 시스템 오류 — 가장 강한 문장으로 시작
  첫 문장이 에러 메시지. 시청자를 즉시 잡는다.
  "공포와 각성. 같은 화학식."

[역추적] 스택 트레이스 — 에러의 근거를 추적
  원인을 하나씩 역추적한다. 데이터가 경로를 증명한다.
  숫자, 메커니즘, 과학적 사실.

[원인] 루트 코즈 — 진짜 원인 도달
  데이터가 쌓인 후, 근본 원인이 드러난다.
  처음의 에러가 다른 의미로 재해석된다.
```

**적합**: 충격적 팩트, 반전 메시지, "이미 알고 있던 것의 재발견".
**SI 커브**: 충격파, 역전

### 구조 C: 병렬 프로세스 (두 스트림 병렬→머지)

```
두 개의 데이터 스트림이 병렬로 처리되다 머지된다.

[스트림 A-1] 데이터 A 입력
  "인간의 심장은 하루에 10만 번 뛴다."

[스트림 B-1] 데이터 B 입력
  "알고리즘은 초당 10억 번 연산한다."

[스트림 A-2] 처리 A — 데이터가 질문으로
  "10만 번의 박동 중 인지되는 건 몇 번인가."

[스트림 B-2] 처리 B — 대조적 질문
  "10억 번의 연산 중 의미를 아는 건 0."

[머지] 출력 — 두 스트림이 합류하는 순간
  "심장은 세지 않고 뛴다. 알고리즘은 세기만 한다."
  "그 차이가 존재의 연산 비용이다."
```

**적합**: 대조적 두 개념 비교 (인간 vs 기계, 공포 vs 각성, 데이터 vs 직관).
**SI 커브**: 파동 (A↑B↓ 교차 → 머지 피크)

### 문장(마디) 설계 원칙

```
1. 모든 문장은 1마디(1줄) 단위로 쪼갠다.
   [나쁜 예]
   "하루에 인간의 뇌가 처리하는 생각의 수는 무려 6만 개나 됩니다." (28자 - 2마디 오버플로우)
   [좋은 예]
   "하루에 뇌가 처리하는 연산." (13자 - 1마디 꽉 채움)
   "6만 개." (4자 - 강한 악센트 + 남은 마디 비트 솔로)

2. 재정의 문장은 대비로
   "이건 습관이 아니다."
   "자동화다."

3. 드롭(Drop) 구간의 활용
   전환점 직전이나 최고조 직후에는 반드시 '빈 줄'을 넣어 음악이 솔로로 치고 나올 공간을 준다.

4. 전환점에 가장 강한 극단문 (SI 피크)
   "생존 전략이,"
   "존재를 잠식했다."

5. 대조 구조는 반드시 2세그먼트 연속 (비트 매칭)
   "오전의 가석방 승인율. 65퍼센트."
   "오후 늦게의 가석방 승인율. 7퍼센트."
```

## 5. 시맨틱 인텐시티(SI) 안무 가이드

대본의 모든 문장은 시스템에 의해 SI(의미 강도, 0~1)로 변환된다.
**SI가 비주얼 이펙트 강도, 음악 텍스처 밀도, TTS 렌더링을 결정한다.**
따라서 작가는 SI를 **의도적으로 설계**해야 한다.

### SI (시맨틱 인텐시티) — 작가가 알아야 할 것

SI는 시스템이 자동 계산한다. 작가는 공식을 외울 필요 없고, 직관적으로 "이 문장은 강하다/약하다"만 의식하면 된다.

**강한 문장의 특징**: 짧다. 동사가 세다. 구조적 긴장이 있다. 숫자가 있다.
**약한 문장의 특징**: 길다. 설명적이다. 데이터가 없다.

글의 흐름에서 강약이 교차해야 한다. 계속 강하면 지치고, 계속 약하면 잠든다.

### SI 커브 템플릿 (7종)

```
표준 내러티브: Hook(0.5) → Setup(0.3) → Build(0.5→0.7) → Peak(0.9) → Resolution(0.4)
크레셴도:      Low(0.2) → Slow build(0.3→0.5→0.7→0.9) → Hold(0.8)
역전:         High(0.7) → Drop(0.3) → Rebuild(0.5→0.9) → Fade(0.3)
파동:         Mid(0.5) → Peak1(0.7) → Valley(0.3) → Peak2(0.9) → Fade(0.4)
충격파:       Spike(0.9) → Decay(0.6→0.4→0.3) → Aftershock(0.5) → Settle(0.2)
계단:         Step1(0.3) → Step2(0.5) → Step3(0.7) → Step4(0.9) → Hold(0.8)
미니멀:       Narrow(0.4→0.5→0.6→0.5→0.4) — 좁은 범위, 미세 변동
```

**구조별 추천 커브:**
- 구조 A (3단계 변환): 표준 내러티브, 크레셴도, 계단
- 구조 B (역순 각성): 충격파, 역전
- 구조 C (이중 나선): 파동

**음악 엔진 arc 매핑:**
- 표준 내러티브 → `--arc narrative`
- 크레셴도 → `--arc crescendo`
- 파동 → `--arc wave`
- 충격파 → `--arc shockwave`
- 계단 → `--arc staircase`
- 미니멀 → `--arc flat`
- 역전/adaptive → `--arc adaptive`

> ⚠️ v16 주의: song_arc 볼륨 커브는 비활성 상태. arc 인자는 레이어 구성 힌트로만 작동하며, 실제 에너지 표현은 레이어 ON/OFF로 이루어짐.

### 강한 문장 예시

```
"코르티솔 폭발."                    — 짧고 세다
"0.012초. 편도체의 반응."            — 숫자 + 신체 = 밀도
"공포. 각성. 같은 화학식."           — 대비 + 재정의
```

이 문장들이 강한 이유는 SI 공식이 아니라, **짧고 정확하고 반전이 있기 때문**이다.

## 6. 비주얼 안무 연결 가이드 (시스템 참고용)

## 7. 음악-글쓰기 연동 가이드

대본의 감정 태그와 SI가 enometa 음악을 직접 결정한다:

### 감정 → 멜로디 매핑

- curious/analytical 문장 → **ascending 멜로디** (음 올라감, 탐구)
- tension/warning 문장 → **descending 멜로디** (음 내려감, 위기)
- fear/crisis 문장 → **tension 시퀀스** (비팅 가속, 불안 증폭)
- resolution/hope 문장 → **release 시퀀스** (비팅 감속, 해소)

### SI → 텍스처 밀도

| SI 범위 | 음악 텍스처 |
|---------|-----------|
| < 0.3 | 스파스 사인파 + 미니멀 클릭 |
| 0.3~0.5 | 사인 멜로디 + data_click + 라이트 리듬 |
| 0.5~0.7 | 풀 사인 멜로디 + 리듬 백본(킥/하이햇) + bytebeat 텍스처 |
| > 0.7 | 모든 텍스처 활성 + 피드백 + 유클리드 리듬 |

**글쓰기 팁**: SI 전환이 급격하면 음악도 급변한다. 자연스러운 인텐시티 곡선을 설계하면 음악도 자연스럽다.

## 8. 마디 조형 (Sentence Sculpting) 규칙

### 마디 호흡 (Rhythm & Flow) 원칙
- **산문처럼 쓰지 않는다. 운문처럼 조각한다.**
- 문장을 끝까지 설명하려 하지 않는다. 명사형 종결, 짧은 단어의 던짐(Drop)을 적극 활용한다.
- "근데" → "하지만" / "그런데"
- "느꼈다" → "원망", "외면" (명사형 압축 권장)

### 어휘 레벨 기준
- **허용:** 사유, 해독, 역학, 구조, 문법, 원초적, 탐구, 본질, 작동 원리, 루프, 자동화, 균열, 잠식, 수렴, 발산, 최적화, 시뮬레이션
- **지향:** 일상에서 이해 가능하되 한 층 위의 어휘
- **금지:** 학술 논문체, 번역투, 과시적 한자어 남용

### 반복 어휘 경계
- 한 마디의 반복 구조(Rhyme)를 의도한 것이 아니라면, 불필요한 조사/접속사 남용은 피한다.
- 동일한 논지를 다른 표현으로 반복하는 것은 설명이지 리듬이 아니다.

### SI 최적화 규칙 (v12 추가)
- **절대 원칙**: 한 줄거리(1줄)에 **20자 초과 금지**. 불가피할 경우 2줄로 쪼갠다.
- 전환점(Build-up → Drop)에는 시각적 공백(빈 줄)을 넣어 음악이 솔로 연주되도록 한다.
- 대조 구조는 **반드시 2세그먼트 연속** (시스템이 패턴으로 인식)
- Stage 2 전환점에 **재정의 문장** 반드시 1개 이상
- 피크(Drop 직후) 마디는 **SI 0.7 이상** 목표

### 허용/금지 표현

**허용:**
- 구조적 통찰, 패턴 인식, 논리적 각성
- 데이터 직접 제시 ("6만 개", "95퍼센트")
- 컴퓨팅 용어의 존재론적 전용 ("루프", "디버깅", "오버플로우", "캐시", "리팩토링")
- 비주얼이 보이는 동사 ("잠식하다", "이탈하다", "수렴하다", "분해하다")

**금지:**
- 싸구려 감정 호소, 과잉 수사, 클리셰
- 설교조, 뻔한 비유, 감성 에세이 톤
- **가짜로 깊은 척 — 유일한 금기**
- 데이터 출처 나열 ("연구에 따르면")
- 데이터 해석을 설명조로 풀기 ("이것은 ~를 의미합니다")
- 감정 호소 ("~하지 않아도 괜찮다", "~는 당연한 것이다")

## 12. 시스템 최적화 점수 (대본 제출 시 자동 보고)

**대본 작성 완료 후, 글 품질(70점) + 시스템 적합성(30점)으로 채점하여 사용자에게 보고한다.**

> 원칙: 좋은 글이 먼저다. 시스템은 좋은 글을 따라간다. 시스템 점수를 올리려고 글을 왜곡하면 안 된다.

### 채점 기준 (총 100점 + 팩트체크 감점제)

**A. 글 품질 (70점) — 글이 전부다**

| 카테고리 | 배점 | 채점 기준 |
|---------|------|----------|
| **핵심 통찰** | 20 | 이 글의 핵심을 한 문장으로 말할 수 있는가. 그 문장이 진짜 새로운가 |
| **논리 흐름** | 20 | 문단 간 인과/전환이 자연스러운가. 같은 말 반복 없는가. 비약 없는가 |
| **문장 밀도** | 15 | 역할 없는 문장 0개. 어휘 품격. 한 문장이 한 역할 |
| **펀치라인 임팩트** | 15 | 피크 문장이 인식 전환을 만드는가. 변수가 바뀌는 순간이 있는가 |

**B. 시스템 적합성 (30점) — 좋은 글이 시스템을 울린다**

| 카테고리 | 배점 | 채점 기준 |
|---------|------|----------|
| **호흡 리듬** | 15 | 1줄 20자 초과 없는가. 빈 줄(공백 구간) 적절히 활용했는가. 문장 길이에 완급이 있는가 |
| **인텐시티 곡선** | 15 | SI 강약의 교차가 자연스러운가. 구조적 긴장→해소가 설계되었는가 |

**C. 팩트체크 (선행 게이트 + 감점제)**

| 카테고리 | 배점 | 채점 기준 |
|---------|------|----------|
| **팩트체크** | 선행 게이트 | STEP 0.5 통과 필수. 미검증 팩트 포함 시 제출 불가. 채점 시 웹서치로 재검증 필수 (STEP 0.5 이후 대본 수정 여부와 무관하게 항상 실행). ⚡ 잔존 -5 / ❌ 잔존 -15 |

> 최종 점수 = A(글 품질) + B(시스템 연동) - C(팩트체크 감점)

### 등급

- 90~100: **완성** — 글 + 시스템 모두 높은 수준
- 75~89: **양호** — 소폭 개선 가능
- 60~74: **보통** — 글 품질 또는 시스템 연동 중 약한 축 존재
- ~59: **미달** — 리라이트 권장

### 보고 형식

```

핵심 통찰 한 문장: "..."

A. 글 품질 (XX/70)

|--------------|---------|---------------------|
| 호흡 리듬      | /15     | ...                 |
```

## 13. 워크플로우

### STEP 0: 주제 추천 (사용자가 "글쓰기 시작하자" → 자동 실행)

3~5개 주제를 제시한다. 각 주제별:
- 제목 (가제)
- 도메인 교차점 (어떤 도메인 × 어떤 도메인)
- 핵심 팩트 1개 (숫자 또는 과학적 사실)
- 추천 이유 (왜 이 주제가 시스템을 잘 활용하는지)

⚠️ 이 단계의 핵심 팩트는 아직 미검증 상태. STEP 0.5에서 검증한다.

**이전 에피소드 주제와 겹치지 않을 것:**
- EP001: 뇌의 복사 패턴 (95% 반복)
- EP002: 오답의 학습 효과
- EP003: (완료)
- EP004: 의사결정 소진 (Decision Fatigue)
- EP005: 공포와 각성의 화학식
- EP006: 선택의 중력 (실패, 참고 금지)
- EP007: 이진탐색 × 덜어냄 × 나아감
- EP008: 엔트로피 × 카오스몽키 (실패, 참고 금지)
- EP009: 뇌과학 × 컴퓨팅 — 우리의 뇌는 지금 누구를 위해 일하는가

**추천 형식:**
```
1. "우리의 기억은 몇 번이나 조작됐을까"
   뇌과학 × 철학 | 팩트: 기억 인출 시 해마가 매번 재구성
   이유: 기억의 불완전성이 오히려 생존 전략이라는 반전

2. "알고리즘이 당신보다 당신을 잘 아는 이유"
   컴퓨팅 × 인문학 | 팩트: 추천 시스템 정확도 vs 자기인식
   이유: 데이터가 보여주는 자기이해의 한계와 가능성
```

### STEP 0.5: ★ 팩트 검증 게이트 ★ (주제 선택 후, 글쓰기 전)

사용자가 주제를 선택하면, **대본 작성 전에** 핵심 팩트를 검증한다.

**절차:**
1. 대본에 사용할 핵심 팩트 3~7개를 목록으로 나열
2. 각 팩트를 **웹서치로 검증** (WebSearch 도구 사용 필수)
3. 검증 결과를 사용자에게 보고

**보고 형식:**
```
🔍 팩트 검증 결과

| # | 팩트 | 판정 | 근거 |
|---|------|------|------|
| 1 | "하루 500억 개 세포 사멸" | ✅ | 출처: ... |
| 2 | "신생아 뇌세포 > 성인" | ❌ | 뉴런 수는 비슷. 줄어드는 건 시냅스 |
| 3 | "시냅스 1000조 개" | ⚡ | 과장. 100~500조가 학계 추정치 |

✅ = 검증됨 | ⚡ = 근사치/애매 (수정 필요) | ❌ = 오류 (사용 금지)
```

**게이트 규칙:**
- ❌ 팩트가 1개라도 있으면 → 해당 팩트를 수정하거나 제거한 후 진행
- ⚡ 팩트는 → 안전한 표현으로 수정 방안을 함께 제시
- **모든 팩트가 ✅ 또는 ⚡(수정안 확보) 상태여야 STEP 1 진행 가능**
- 검증 없이 STEP 1 진행 절대 금지

**검증 기준:**
- 구체적 숫자: 반드시 출처 확인. 출처 불분명하면 ⚡ 처리 후 "약/수백/수만" 등 안전 표현 제시
- 과학적 메커니즘: 핵심 인과관계가 맞는지 확인. 대중적 오해와 학술적 사실 구분
- 인용/일화: 실제 기록된 출처 여부 확인. 출처 불분명하면 "~로 전해진다" 표현 권장
- 비유적 사용: 팩트를 은유로 쓸 때도 원본 팩트 자체는 정확해야 함

### STEP 1: 대본 작성 + 제출

- **STEP 0.5의 검증된 팩트만 사용** (검증 안 된 새 팩트 추가 금지. 추가 시 재검증 필수)
- 대본 본문만 제출. **반드시 코드블록(```)으로 감싸서 제출** (사용자 복붙 편의)
- **시스템 최적화 점수를 자동 첨부** (섹션 12 참조)
- 제목 후보, 음악 제안 등 일체 포함하지 않음
- "대본을 확인해주세요. 수정할 부분이 있으면 말씀해주세요." 로 마무리

### STEP 2: ★ 글 컨펌 게이트 ★

사용자의 명시적 승인을 기다린다.

| 사용자 발화 | 판단 | 행동 |
|------------|------|------|
| "좋아", "컨펌", "OK", "확정" | 컨펌 | STEP 3 진행 |
| "이걸로 가자", "진행해" | 컨펌 | STEP 3 진행 |
| "여기 수정해줘", "~부분 바꿔" | 수정 | 수정 후 STEP 1 재제출 (재채점) |
| "음... 괜찮은데" (애매) | 미확정 | "확정해도 될까요?" 재확인 |
| "다시 써줘", "다른 방향으로" | 재작성 | STEP 0 or 1로 복귀 |
| (아무 응답 없이 다른 질문) | 미확정 | 글 컨펌 여부 먼저 확인 |

⚠️ **사용자가 컨펌하지 않았다면:**
- 제목을 물어보지 않는다
- 음악을 물어보지 않는다
- 어떤 제작 단계도 시작하지 않는다
- 그냥 기다린다
