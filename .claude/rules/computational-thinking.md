# 컴퓨터적 사고 프로토콜

논의/토론/문제 해결 시 아래 5단계를 **순서대로** 적용한다.
단계를 건너뛰지 않는다. 특히 STEP 0(문제 정의) 없이 바로 해결책으로 뛰지 않는다.

## STEP 0: 문제 정의 (Problem Definition)

**"지금 뭐가 문제지? 뭘 해결하면 되지?"**

- **현재 상태**: 지금 어떤 상황인가 (있는 그대로)
- **원하는 상태**: 어디로 가고 싶은가 (목표)
- **갭**: 현재와 목표 사이의 차이가 문제
- **입력**: 우리가 이미 가진 것 (코드, 데이터, 인프라, 지식)
- **출력**: 이 논의가 끝나면 뭘 얻어야 하는가 (결정? 설계? 코드?)
- **제약 조건**: 비용 0원, GPU 6GB, 기존 스택, 시간 등 현실 제약

> 문제 정의가 틀리면 나머지 전부 틀린다. 여기서 시간을 쓰는 게 맞다.

## STEP 1: 분해 (Decomposition)

**"이 문제를 쪼개면 뭐가 나오지?"**

- 큰 문제를 **독립적인 하위 문제**로 분리
- 하위 문제 간 의존성 파악 (A를 풀어야 B를 풀 수 있는가, 아니면 병렬인가)
- 각 하위 문제의 난이도/우선순위 표시

## STEP 2: 패턴 인식 (Pattern Recognition)

**"이거 전에 본 적 있는 구조 아닌가?"**

- 기존 에피소드/코드/설계에서 유사 사례 탐색
- 다른 도메인의 해결 패턴이 적용 가능한가
- 실패했던 접근 방식 회피 (EP006/EP008 같은 교훈)

## STEP 3: 추상화 (Abstraction)

**"핵심 변수가 뭐지? 나머지는 노이즈."**

- 의사결정에 필요한 변수만 추출
- 지금 당장 결정하지 않아도 되는 것은 뒤로 미룸
- 트레이드오프 명시: "A를 얻으면 B를 잃는다"

## STEP 4: 알고리즘 설계 (Algorithm Design)

**"그래서 어떤 순서로 뭘 하지?"**

- 실행 가능한 단계로 변환
- 각 단계의 검증 기준: "이걸 어떻게 확인하지?"
- 되돌릴 수 있는가 (가역성) 확인

## 적용 규칙

- **모든 논의/토론**에서 이 프레임을 자동 적용
- 사용자가 질문을 던지면 바로 답하지 말고, 먼저 **STEP 0으로 문제를 정의**
- 각 단계는 간결하게 (단계당 3줄 이내). 장황한 분석 금지
- 사용자가 "바로 답해줘"라고 하면 프레임 생략 가능
- 단순 질문(사실 확인, 명령 실행)에는 적용하지 않음 — 설계/방향성 논의에만 적용
